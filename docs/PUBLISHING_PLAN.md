# Aura-Video — Native Publishing Integration Plan

Status: **DRAFT — awaiting your review.** Part A of the publishing-unfreeze request. No implementation code has been written. Do not start Part B until this memo and `docs/SETUP_GUIDE.md` are confirmed and you've handed over credentials (checklist at the end of the setup guide).

This replaces the original plan of turning `publishing_enabled` on top of the existing Upload-Post third-party aggregator. Instead: native integrations with YouTube Data API v3 and Instagram's Graph/Content Publishing API, built behind a `PublishingProvider` interface. `app/services/upload_post.py` stays in the tree as a disabled legacy provider — not deleted, not wired to anything by default.

Every platform-behavior claim below was checked against current official documentation in July 2026 (Google for Developers, Meta for Developers) rather than carried over from the brief as given — where the brief's numbers were stale, I've corrected them and said so explicitly.

## 0. What already exists (read before deciding anything below)

Confirmed by direct inspection, not assumption:

- `publishing_enabled` (`config.example.toml` §`[features]`, read in `app/config/config.py`) is the master switch. `orchestrator.approve_and_publish()` (`app/agents/orchestrator.py:1137-1179`) checks it and short-circuits to `APPROVED` with no publish thread when false — this is the only gate that needs new logic; the state machine itself doesn't change.
- **Final Review (`AWAITING_HUMAN_APPROVAL → APPROVED`) is the one and only path that calls `Publisher.publish()`**, unconditionally, regardless of Approval Mode (`manual`/`automatic`). Approval Mode only ever affects the earlier script-approval gate. This plan does not touch that invariant.
- `VideoProject.publish_package` (JSON) already holds SEO title options, description, tags, category, per-platform caption variants, and thumbnail candidates, generated by `Publisher.prepare()` (`app/agents/publisher.py`). It's editable at Final Review via `PATCH /projects/{id}/metadata`, which records `human_edits` for the learning loop — this plan extends that structure, doesn't replace it.
- `VideoProject.published_posts` (JSON list) and `VideoProject.published_at` are the two fields every downstream system already keys off: `PerformanceAnalyst.extract_youtube_video_id()` walks `published_posts` looking for a video ID or YouTube URL; the originality engine's fingerprint-retention rule (`app/services/project_deletion.py`) keys off `published_at is not None`. **Any native publish result must write into these same fields, or both systems silently break for natively-published videos.** New tables (§3 below) are additive detail; they are not a replacement for this.
- Background jobs = daemon threads spawned from `orchestrator.py`, not a queue. Progress is pushed live via `ws_manager.broadcast_event()` and durably logged via `AgentEvent` rows (`BaseAgent.log_event()`, the single call site that does both). This is the pattern new publish jobs will follow — no new job framework.
- No OAuth, no token encryption, no per-user credential storage exists anywhere in this codebase today. All current secrets are plaintext in `config.toml` (git-ignored, developer-supplied API keys, never returned to the frontend). This is a first for the app — designed from scratch below (§2).
- Manual "Mark as Published" (`POST /projects/{id}/mark-published`) is fully implemented today and stays exactly as-is for any platform without a native integration, or as a fallback if a native publish needs manual correction.

## 1. Platform realities — verified, and where the brief needs correcting

### YouTube

**Confirmed as stated:**
- **API audit gates public visibility.** Since July 28, 2020, any video uploaded via `videos.insert` from an unaudited API project is locked to `private` and cannot be switched to `public`/`unlisted` by the uploader — only Google, via a passed audit, lifts this. [YouTube Help: Videos locked as private](https://support.google.com/youtube/answer/7300965). The app must never let this surprise you: see §4.
- **OAuth Testing-mode refresh tokens expire after 7 days.** Confirmed for External-type OAuth apps left in "Testing" publishing status — this is separate from and unrelated to the API audit above. [Google Developer forums thread](https://discuss.google.dev/t/oauth2-refresh-token-expiration-and-youtube-api-v3/160874); [Napkyn writeup](https://www.napkyn.com/blog/youtube-api-refresh-token-expiring-fix). Fix is "Publish" the OAuth consent screen (Google's own verification/publishing flow, distinct from the YouTube API audit) — no App Review needed for a single-user internal tool since you'll add yourself as a test user, but Testing-mode's 7-day refresh-token ceiling makes that impractical for daily use. Plan: detect expired-refresh-token failures specifically and surface "reconnect your YouTube account" rather than a generic error; document publishing the consent screen in the setup guide as the real fix.
- **No API to pin a comment.** `commentThreads.insert` posts a top-level comment; nothing in the Comments API pins it. [Confirmed via `developers.google.com/youtube/v3/docs/comments`]. Plan: post the first-comment text automatically after upload if provided, and label it "posted — pin manually" in the UI. Never claim automated pinning.
- **`thumbnails.set` requires phone-verified channel.** This isn't a special "verified checkmark" — it's YouTube Studio's "phone verification" (Studio → Settings → Channel → Feature eligibility → "Features that require phone verification"), a free one-time step, capped at 2 channels per phone number per year. [YouTube Help: Verify your account](https://support.google.com/youtube/answer/171664). If unverified, the app must not silently drop the thumbnail — it must explain the exact missing step.
- **Scheduling uses native `status.publishAt` + `status.privacyStatus = "private"`.** YouTube auto-flips the video to public at that timestamp. Confirmed via `videos/insert` reference docs.

**Correction to the brief — quota mechanics changed:** The brief's figure (`videos.insert` costs ~1,600 units from the shared 10,000-unit pool, ⇒ ~6 uploads/day) is **out of date**. As of the current API quota documentation (`developers.google.com/youtube/v3/determine_quota_cost` and `.../guides/quota_and_compliance_audits`, both fetched directly), `videos.insert` (like `search.list`) now bills against its **own dedicated bucket of 100 calls/day**, separate from the shared 10,000-unit pool used by everything else (`videos.update`, `thumbnails.set`, `commentThreads.insert` each still cost ~50 units from the shared pool). Net effect: **the default ceiling is ~100 uploads/day, not ~6** — much less pressure than the brief assumed, though still a hard cap worth tracking and surfacing in the UI, and still something to mention in the audit-request form as future-proofing. I could not find an official changelog entry pinning the exact date of this change, only convergent current documentation and third-party trackers citing a mid-2026 rollout — **re-verify this number against the live quota calculator right before implementing**, since it's the kind of platform change that could move again.

### Instagram

**Confirmed, with a resolved discrepancy:** the Content Publishing Limit reference page (`developers.facebook.com/docs/instagram-platform/instagram-graph-api/reference/ig-user/content_publishing_limit/`) states the current default cap explicitly: **50 published containers per rolling 24-hour window** (`quota_duration` = 86400s), not the 25 the brief cited (25 was the original 2018-era default; several current third-party blogs still repeat it incorrectly). Reels, images, and carousels all count against the same bucket. Query the `content_publishing_limit` edge on the IG user to read live usage rather than hardcoding — plan to poll and cache it, and enforce app-side before attempting a publish call that would exceed it.

- **Account type: Professional (Business or Creator) required, but recommend Business.** Both product docs (`instagram-api-with-facebook-login` and the newer `instagram-api-with-instagram-login`) require an "Instagram professional account." In practice, third-party API publishing access is explicitly a Business-account-oriented feature (Creator accounts are steered toward Meta Business Suite instead), so **convert to Business, not Creator**, in the setup guide.
- **Two publishing paths, confirmed both support Reels container publishing:**
  1. **Classic — "Instagram API with Facebook Login" (IG linked to a Facebook Page).** Requires a Facebook Page, Page Publishing Authorization if applicable, `graph.facebook.com`, and supports the `upload_type=resumable` chunked-upload alternative to the public-URL fetch.
  2. **Newer — "Instagram API with Instagram Login" / Business Login for Instagram.** No Facebook Page or Facebook account in the loop; authenticates directly against Instagram; uses `graph.instagram.com`; fewer permissions (`instagram_business_basic`, `instagram_business_content_publish`); **does not support resumable upload** — public-URL fetch only.
  - **Recommendation: Instagram API with Instagram Login**, for this single-owner app. It removes the Facebook-Page-linking step entirely (one less account entanglement, one less thing to misconfigure), and the resumable-upload feature it lacks is irrelevant anyway — our `video_url` fetch design (below) is public-URL-based regardless of path, so the "missing" feature costs nothing here.
- **App Review vs. Development mode — confirmed, use Development mode.** Meta's own docs: "If your app will only be used by app users who have a role on the app itself, App Review is not required." Adding your own Instagram/Meta account with a role (Admin/Developer/Tester) on the app in Development mode lets it call `instagram_business_content_publish` against your own account with zero App Review. App Review only becomes necessary if this ever needs to publish to an account that doesn't have a role on the app — i.e., if this stops being a single-owner tool. Document that trigger explicitly in the setup guide so future-you knows when to revisit it.
- **The public-URL problem is real and architectural, confirmed.** Both Instagram publishing paths create a media container via `POST /{ig-id}/media` with a `video_url` that **Meta's servers fetch directly** — "media must be hosted on a publicly accessible server at the time of the attempt." A purely local dev server has no such URL. See §5 for the `MediaHosting` abstraction.
- **No native Instagram scheduling API.** Confirmed — Instagram's Content Publishing API has no `publish_at`-equivalent field; the only way to "schedule" is to hold the publish server-side and fire it at the right time via our own scheduler (APScheduler, already in use for hourly performance checks — same mechanism, new job).
- **Container status polling:** `GET /{container-id}?fields=status_code` → `EXPIRED` (containers auto-expire in 24h if never published) / `ERROR` / `FINISHED` / `IN_PROGRESS` / `PUBLISHED`. Meta's own guidance: poll roughly once a minute, for up to ~5 minutes, before treating it as stuck.

## 2. Provider architecture

```
app/services/publishing/
  __init__.py
  base.py            # PublishingProvider ABC
  youtube.py          # YouTubeProvider
  instagram.py         # InstagramProvider
  upload_post.py        # thin adapter wrapping the existing service, DISABLED by default
  media_hosting.py       # MediaHosting abstraction (§5)
  token_store.py         # encrypt/decrypt, refresh scheduling
```

`PublishingProvider` interface (exact methods from the brief, all async-friendly since they're called from background threads):

```
connect(auth_code, redirect_uri) -> ConnectedAccount
disconnect(account_id) -> None                    # revokes token remotely, deletes local row
validate_metadata(publish_package, platform_overrides) -> list[ValidationError]
publish(project, metadata, media_refs) -> PublishResult   # PublishResult carries platform_video_id/url or error
check_status(attempt_id) -> PublishStatus
account_info(account_id) -> AccountInfo            # channel/account name, avatar, connection health
remaining_quota(account_id) -> QuotaInfo             # e.g. {"used": 12, "limit": 100, "resets_at": ...}
```

`upload_post.py`'s adapter implements the same interface around the existing `UploadPostService` (its real, working `upload_video`/`check_status` calls, §1 of the codebase report) but its provider registration is gated behind an explicit, separate flag (`config.features.legacy_upload_post_enabled`, default `false`) — distinct from the new per-platform `publishing_enabled` semantics below, so flipping general publishing on never silently reactivates the old aggregator.

**Flipping `publishing_enabled` semantics (per your instruction):** `true` activates the publish panel in the UI; each individual platform additionally requires a connected, healthy account (checked via `account_info()`) before its toggle is selectable. A project can still reach `APPROVED`/"Ready to Publish" with zero platforms connected — connecting is not a precondition for approval, only for actually publishing. The hard human-approval gate (Final Review) is untouched — nothing here creates a second path to `Publisher.publish()`.

## 3. OAuth flows, token storage, and scheduling

**OAuth 2.0 authorization-code flow, both platforms**, state-CSRF-protected:

- `GET /api/v1/oauth/{provider}/start` — generates a signed `state` (project-scoped or account-scoped, short TTL), redirects to the platform's consent screen with the minimum scopes (§ setup guide).
- `GET /api/v1/oauth/{provider}/callback` — validates `state`, exchanges the auth code for tokens, encrypts and stores them, redirects back into Settings with a success/failure banner.
- Settings gets a "Connect"/"Disconnect" card per platform: connected account/channel name + avatar, token health (`valid` / `expiring soon` / `reconnect needed`), Disconnect (revokes remotely where the platform supports revocation, then deletes the local row regardless).

**Token storage — new table `platform_credentials`:**

| column | notes |
|---|---|
| `id` | PK |
| `platform` | `youtube` \| `instagram` |
| `account_label` | channel/account display name, refreshed on connect and periodically |
| `access_token_enc` | Fernet ciphertext |
| `refresh_token_enc` | Fernet ciphertext, nullable (Instagram long-lived tokens self-refresh differently, see below) |
| `expires_at` | UTC |
| `oauth_scope` | space-joined, for audit/debugging without needing the token itself |
| `status` | `connected` \| `expiring` \| `reconnect_required` |
| `created_at`, `updated_at` | |

Encryption: Fernet symmetric, key from `TOKEN_ENCRYPTION_KEY` env var. **Publishing features refuse to start (clear startup error, not a silent no-op) if this env var is missing** — this is the one new "hard requirement" env var in an app that's otherwise all-TOML-config, called out prominently in the setup guide. Never logged: `BaseAgent.log_event()`/`AgentEvent` payloads and `project.json` are covered by an explicit denylist check in the token-store module (a token string can never be interpolated into any log/event/manifest write — enforced by only ever passing `account_label`/`status` to those call sites, never the credential row itself).

**Refresh scheduling, per platform:**
- **YouTube:** standard Google OAuth refresh-token flow; token refreshed automatically ~5 minutes before `expires_at`. In Testing-mode OAuth apps (§1), Google itself invalidates the refresh token after 7 days regardless of use — refresh will start failing with `invalid_grant`; that specific error is mapped to `status = reconnect_required` (not a generic failure), and Settings shows "Reconnect YouTube" with a one-line explanation, not a stack trace.
- **Instagram:** short-lived token (1 hour) from the OAuth callback is immediately exchanged for a long-lived token (~60 days) in the same request. Long-lived tokens are refreshable via a dedicated refresh endpoint **as long as the current token is at least 24 hours old and not yet expired** — a background job (APScheduler, daily) refreshes any Instagram token within 10 days of expiry. If refresh fails (token already expired, account disconnected the app, etc.), same `reconnect_required` UI treatment as YouTube.

## 4. Publishing flow, data model, and pre-audit UX

**New tables:**

```
publish_targets
  id, project_id, platform, status (Ready|Scheduled|Publishing|Published|Failed),
  scheduled_at (nullable), platform_video_id (nullable), url (nullable),
  error (nullable), attempt_count, created_at, updated_at

publish_attempts
  id, publish_target_id, timestamp, request_summary (JSON, secrets stripped),
  response_summary (JSON), error (nullable), retry_of_attempt_id (nullable)
```

`publish_targets` is per project × platform — this is what makes "YouTube succeeded, Instagram failed" representable and honestly shown in the UI (two independent rows, two independent statuses) rather than one project-level publish flag. `publish_attempts` is the full audit trail per target, including every retry, and is the source for the Publishing History timeline (§ UI, Part B).

**Status flow (project-level, unchanged) vs. target-level (new):** `APPROVED` already means "Ready to Publish" today (§0). A project at `APPROVED` with `publishing_enabled=true` gets a `publish_targets` row created per selected platform, defaulting to `Ready`. Clicking Publish Now/Schedule transitions the relevant target(s) to `Publishing`/`Scheduled`; the project's own `VideoProject.status` only moves to `PUBLISHING` once at least one target is actively publishing, and to `PUBLISHED` once every selected target has reached a terminal state (`Published` or `Failed` after exhausting retries) — matching the existing enum, no new project-level statuses needed.

**Pre-audit YouTube UX (the brief's explicit ask):** every YouTube publish, until you've told the app the audit passed (a simple Settings toggle/flag, since there's no API to query audit status), uploads as `privacyStatus=private` and the UI shows a persistent, non-alarming banner on that publish target: "Uploaded — locked to Private until your YouTube API audit is approved. [What does this mean?] [Request the audit]" linking to the setup guide's audit section. This is never a failure state — the upload succeeded; visibility is what's pending.

**Resumability:**
- **YouTube:** resumable upload protocol (chunked, 256KB-aligned), retrying with exponential backoff (1s → 2s → 4s → ... capped, ~5 attempts) on 5xx/network errors, honoring `Retry-After` when present. The resumable session URI is valid ~7 days, so a crash mid-upload can resume from the last committed byte on next attempt rather than restarting — this is what satisfies "pulling the network cable never duplicates a post."
- **Instagram:** no resumable protocol on the public-URL path (that's a Facebook-Login-only feature we're intentionally not using, §1) — instead, idempotency comes from the container model itself: creating a container doesn't publish anything, so a crash before `media_publish` is called just leaves an orphaned container that expires harmlessly in 24h. Status polling after `media_publish` is called (once/minute, ~5 minutes) with a hard timeout; if the app crashes mid-poll, on restart it re-queries the container's `status_code` before deciding to retry, so it never blindly re-publishes a container that actually finished.

**Retry classification:** transient (network errors, 5xx, 429/rate-limit — respect `Retry-After`) → automatic exponential backoff. Permanent (4xx validation errors, policy rejection, `invalid_grant`/auth failure) → no automatic retry, human-readable mapped message, manual Retry button in the UI. A small error-code → plain-English table ships in the provider modules (e.g. YouTube `uploadLimitExceeded` → "Daily upload quota reached, resets at midnight Pacific"; Instagram error code 9 → "Instagram's 24-hour publishing limit reached for this account").

**Performance Analyst handoff:** on a successful YouTube `publish()`, the returned video ID is written directly into `publish_targets.platform_video_id` **and** appended into `VideoProject.published_posts` in the same shape `extract_youtube_video_id()` already parses (a `youtube_video_id` key or an embedded `youtube.com/watch?v=` URL) — zero changes needed to `performance_analyst.py` or the hourly `run_performance_checks()` job; verified against the existing extraction logic rather than assumed.

## 5. MediaHosting abstraction (Instagram's public-URL requirement)

```
app/services/publishing/media_hosting.py

class MediaHosting(ABC):
    def stage(local_path) -> HostedMedia          # HostedMedia: public_url, host_key, expires_at
    def release(host_key) -> None                # best-effort delete after Instagram confirms ingestion
```

**Backend (a) — temporary object storage, default now.** Cloudflare R2 (S3-compatible, free tier: 10GB storage / 1M Class A + 10M Class B ops per month, **zero egress fees**, which matters since Meta's fetch is an egress read every single publish). Flow: upload the final video to a private-by-default bucket right before calling `media` create, generate a short-TTL presigned URL (e.g. 2 hours — comfortably longer than Instagram's ~5-minute polling window plus retries), pass that URL as `video_url`, and delete the object once the container reaches `FINISHED`/`PUBLISHED`. A bucket **lifecycle rule auto-deletes anything older than 48 hours** as a safety net for crashes that skip the explicit delete (setup guide covers creating this rule).

**Backend (b) — self-hosted signed expiring URLs, for when the app is eventually deployed.** A `GET /publishing/media/{token}` route, anchored the same way as the existing project-file route (`file_security.resolve_path_within_directory()`, §0), serving the video with a short-lived HMAC-signed token instead of a bucket. Only viable once the app has a public origin — not usable from a laptop behind NAT, which is why (a) is the default today. Stubbed as an alternate `MediaHosting` implementation, not built out until deployment happens (tracked in `docs/REVIEW_FINDINGS.md` per your instruction below).

**Why R2 over S3 proper:** same API surface (boto3-compatible), no egress charge (S3 chagres ~$0.09/GB egress, which is exactly the pattern of this workload — upload once, one big Meta-server download, done), and the free tier comfortably covers a single-owner tool's volume. Documented as a recommendation, not a hard requirement — the abstraction means swapping to actual S3, Backblaze B2, or Wasabi later is a config change, not a code change.

## 6. DB / config changes summary

- New tables: `platform_credentials`, `publish_targets`, `publish_attempts`.
- `config.example.toml`: new `[publishing]` or extended `[features]` section — per-platform sub-keys for OAuth client IDs (client secrets go in env vars, not TOML, since they're now genuinely sensitive OAuth secrets rather than developer API keys — see setup guide rationale), redirect URI base, R2/S3 bucket config, and a `youtube_audit_approved` boolean (manual toggle, since there's no API to query it).
- New required env var: `TOKEN_ENCRYPTION_KEY` (Fernet key, generation instructions in setup guide).
- New optional env vars: R2/S3 credentials (`R2_ACCESS_KEY_ID`, `R2_SECRET_ACCESS_KEY`, `R2_BUCKET`, `R2_ENDPOINT`) if hosting isn't already configured some other way.
- `upload_post.py` config keys (`upload_post_*`) stay exactly as-is, untouched, still readable by the legacy adapter.
